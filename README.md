# CompanionHR

An AI-powered HR compensation assistant: a single-page React app that processes an entire 25,000-row employee dataset **entirely client-side** — no backend, no database. It ships with five analytics views (Payroll Dashboard, Employee Search, Scenario Simulator, Executive Summary, AI Recommendations) plus an AI HR Assistant chat backed by the Claude API.

Live demo: https://yolokimo768.github.io/companion-hr/
Demo video: [demo/companion-hr-demo.mp4](demo/companion-hr-demo.mp4)

## Getting started

```bash
npm install
npm run dev      # local dev server
npm run build    # production build to dist/
```

The app auto-loads the bundled `public/hr_dataset.csv` on first load. You can also upload your own CSV (see [Data ingestion approach](#data-ingestion-approach) for the required columns) or regenerate the synthetic dataset with `python generate_dataset.py`.

## AI architecture

CompanionHR uses AI in two distinct ways, and it's worth being precise about which is which:

**1. Real LLM integration (AI HR Assistant chat)**
The chat tab (`src/views/AIChatAssistant.tsx`) calls the real Anthropic Claude API directly from the browser via the official `@anthropic-ai/sdk` (`src/utils/claudeClient.ts`), using `claude-opus-4-8`. There is no backend proxy — the request goes straight from the user's browser to Anthropic's API using an API key the user supplies themselves (see [Limitations](#limitations-and-future-improvements) for why).

Because sending all 25,000 rows on every message would be slow and expensive, the model is given:
- A compact **data digest** (`src/utils/dataDigest.ts`) — aggregated stats per department × month, per grade, and per office/location (latest month), plus a few notable individual records — injected into the system prompt as baseline grounding.
- A **tool**, `query_employees` (`src/utils/employeeQuery.ts`), that the model can call to search the *full* dataset with arbitrary filters (department, location, grade, salary/performance/attrition/overtime/absence ranges, promotion eligibility, name/ID lookup) whenever a question needs row-level data the digest doesn't cover.

`claudeClient.ts` runs a small agentic loop: it sends the conversation, and if Claude responds with a tool call, the app executes `query_employees` against the in-memory dataset, feeds the JSON results back to Claude, and repeats (up to 5 rounds) until Claude produces a final text answer. This lets the model answer compound, cross-tabulated questions (e.g. "which employees have high performance but low compensation?") by issuing one or more targeted, self-directed queries instead of being limited to whatever was pre-aggregated.

**2. Local, rule-based "AI-style" features (no LLM call)**
Three parts of the app produce AI-sounding output without calling any model:
- **Local query engine** (`src/utils/chatEngine.ts`) — a regex/keyword-based fallback chat engine used when no API key is set, handling a fixed set of query patterns (lookups, rankings, comparisons, thresholds).
- **AI HR Summary** (`src/utils/narrativeSummary.ts`) — a templated, data-driven narrative on the Executive Summary page (payroll trend + drivers, attrition hotspot, promotion pipeline), generated by plain TypeScript logic over the dataset.
- **AI Recommendations** (`src/utils/recommendations.ts`) — the four ranked lists (promotions, retention, hiring, comp adjustments) are deterministic heuristics over fixed thresholds, not model output.

These are fast, free, and fully deterministic, but they are pattern-matching/heuristics, not generative AI — only the chat tab (when connected to Claude) is a true LLM integration.

## Data ingestion approach

- **Source**: a synthetic dataset (`generate_dataset.py`) of 6,250 employees × 4 months (Jan–Apr) = 25,000 rows, with grade-scaled salaries, department-skewed overtime, office/location assignment, monthly absence tracking, and performance/attrition figures correlated the way real HR data tends to be.
- **Parsing**: 100% client-side via PapaParse. `src/hooks/useHRData.tsx` fetches `/hr_dataset.csv` (or a user-uploaded file) and parses it in the browser — the raw CSV never touches a server.
- **Validation**: every row is checked against a fixed set of required columns (`EmployeeID`, `Name`, `Department`, `Grade`, `MonthlySalary`, `OvertimeHours`, `OvertimePay`, `PerformanceRating`, `AttritionRiskScore`, `PromotionEligible`, `Month`); a CSV missing any of them is rejected with an error rather than silently producing broken rows. `Location` and `AbsenceDays` are read if present but are **optional** — a CSV without them still uploads fine (missing locations fall back to "Unspecified", missing absences default to 0), so older or third-party datasets keep working unchanged.
- **Coercion**: each raw (string) CSV row is converted into a strongly-typed `HRRecord` (numbers parsed, `PromotionEligible` normalized to a boolean, `Department`/`Grade`/`Month` cast to their union types) — see `src/types.ts` for the shape.
- **Scoping**: a global month selector *and* office/location selector (both in the header) filter the in-memory dataset down (`filteredRecords`); every view/chart/table reads from this shared, already-scoped array via the `useHRData()` context.
- **Re-ingestion**: uploading a new CSV or clicking "Reload default dataset" replaces the in-memory dataset entirely — there is no merge/append logic and nothing is persisted between page reloads.

## Prompt strategy

The chat's system prompt (`SYSTEM_PROMPT_INTRO` in `src/views/AIChatAssistant.tsx`) is built fresh on every request and combines four things:

1. **Role framing** — tells Claude it's an HR financial/compensation analyst and to speak like one (concise, numbers-first).
2. **Source-of-truth instructions** — explicitly tells the model to ground answers in the data digest and the `query_employees` tool rather than inventing figures, and to call the tool for anything requiring row-level or cross-tabulated data.
3. **Threshold guidance** — since users ask vague things like "high performers" or "low compensation," the prompt tells the model to reason relative to the grade/department averages already present in the digest (e.g. "meaningfully below that grade's avgSalary") unless the user gives exact numbers, rather than guessing arbitrary cutoffs.
4. **Grounding data** — the JSON data digest, plus (when the user's message names a specific person or employee ID) any directly-matching records found by `findEmployeeRecords()`, appended after the instructions.

On top of the system prompt:
- **Conversation history** is capped to the last 12 messages (the initial welcome message is excluded) and sent as a standard multi-turn `messages` array, so the model has short-term context without unbounded growth in request size.
- **Tool descriptions** double as prompting: `query_employees`'s description in `employeeQuery.ts` explicitly tells the model *when* to use it (compound/cross-tabulation questions, listing/counting, name/ID lookups) versus when the digest alone is sufficient, which is what lets the model self-select between "answer directly" and "look something up first."
- **Output rendering**: the model is not told to avoid Markdown — instead, its Markdown output (bold, italics, lists, tables) is parsed and rendered as real formatted UI (`src/utils/markdown.tsx`), so responses look like a native app UI rather than raw text with stray `**`/`#` characters.

## Limitations and future improvements

**Limitations**
- **No shared backend key** — because the app is 100% static/client-side, every visitor must supply their own Anthropic API key (stored only in their own browser's localStorage). There's no way to give all visitors AI access without either exposing a shared key in the public JS bundle (insecure) or adding a backend.
- **Local fallback is limited** — without an API key, the app falls back to a regex/keyword-matched query engine that only understands a fixed set of phrasings; it's noticeably less capable than the real Claude-backed mode and doesn't handle novel phrasing well.
- **Tool-loop cap** — the chat's agentic loop stops after 5 rounds of tool calls; extremely compound, multi-part questions could in theory need more round-trips than that.
- **No prompt caching** — the system prompt (including the full data digest) is resent in its entirety on every message, which is less token/cost-efficient than using Anthropic's prompt caching.
- **No persistence** — uploading a custom CSV is in-memory only; refreshing the page reverts to the bundled default dataset.
- **No authentication** — anyone with the URL can view all employee records. Fine for this synthetic demo dataset, but not suitable as-is for real HR data.
- **Heuristic "AI" features use fixed thresholds** — the AI HR Summary and AI Recommendations lists use hardcoded cutoffs (e.g. 15% under grade median, attrition risk > 75) rather than anything adaptive or model-driven.
- **Single JS bundle** — the production build is one ~840KB chunk (flagged by Vite's build warning); there's no code-splitting yet.

**Future improvements**
- Add a lightweight serverless proxy (e.g. a Cloudflare Worker or Vercel function) so visitors get AI answers without needing their own API key, while keeping the key server-side only.
- Enable Anthropic prompt caching on the system prompt/data digest to cut latency and cost on repeated messages.
- Expand the tool surface (e.g. add dedicated aggregate/trend-comparison tools) so the model relies less on the static digest and more on live, precise queries.
- Persist uploaded datasets locally (e.g. IndexedDB) so a custom upload survives a page refresh.
- Add authentication/access control for any deployment handling real (non-synthetic) HR data.
- Code-split the bundle (e.g. lazy-load the Recharts-heavy dashboard and the Anthropic SDK) to shrink initial load time.
- Stream chat responses token-by-token instead of waiting for the full reply, for a more responsive feel on longer answers.
